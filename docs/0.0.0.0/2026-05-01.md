- Vulkan instance 생성 입력을 최소화하고 GLFW 필수 확장 포인터를 그대로 전달하도록 변경했다.
- 창 생성 전에 GLFW의 Vulkan 지원 여부를 확인하도록 추가했다.
- `assets/data/blocks.json`에 초기 블록 정의를 추가했다.

## Decisions
- raymarch seam의 원인은 `EMPTY` 서브청크 fast-skip 경계 선택 문제로 보고, fast-skip은 유지하되 현재 ray 거리보다 앞에 있는 경계만 skip 대상으로 삼는다.
- CPU의 플레이어 월드 좌표는 장거리 이동 누적 오차를 줄이기 위해 `double`로 유지하고, GPU shader 입력은 현재처럼 `float`로 유지한다.
- 청크 좌표 계산은 CPU의 `double` 플레이어 위치를 기준으로 하고, 실제 렌더링용 좌표는 기존 렌더 경로에 맞춰 `float`로 변환한다.
- 현재 terrain raymarch는 업스케일 테스트를 잠시 제외하고 swapchain과 같은 내부 해상도 스케일 `1.0`을 사용한다.
- 블록별 face texture 선택은 shader 코드 분기보다 GPU 데이터 테이블을 우선 사용한다.
- block definition table의 face 순서는 `+X`, `-X`, `+Y`, `-Y`, `+Z`, `-Z`로 둔다.
- 현재 block definition table은 기존 블록 ID와 texture array 순서에 맞춰 코드에서 생성하고, JSON 기반 자동 생성은 추후 필요 시 검토한다.
- 청크 생성 멀티스레딩은 worker가 CPU 데이터만 만들고 Vulkan 객체는 건드리지 않는 방식으로 제한한다.
- 완료된 청크 적용량은 worker별이 아니라 전체 렌더 프레임 기준 총합으로 제한한다.
- 현재 완료 청크 적용 제한은 프레임당 4개로 둔다.
- 새 청크 slot은 worker 결과가 도착하기 전까지 빈 서브청크로 취급한다.
- 좌하단 임시 프로파일링 표시는 GPU 업로드보다 CPU 청크 갱신 단계를 세분화해서 확인하는 용도로 사용한다.
- `UPDATE TOTAL`은 청크 갱신 전체 시간, `WORLD TOTAL`은 CPU 월드/슬롯 구성 시간으로 본다.
- `SLOT PRUNE`, `GRID SCAN`, `NEW CHUNKS`, `META BUILD`는 CPU 청크 갱신 내부 단계 확인용 임시 지표로 둔다.
- 청크 로딩 범위 이동 최적화는 전체 재구성보다 delta update를 우선 적용한다.
- 기존 physical slot의 metadata는 청크가 계속 로딩 범위 안에 있으면 다시 만들지 않는다.
- slotMap은 작으므로 현재 단계에서는 전체 업로드를 유지하고, subchunk metadata와 cell data만 부분 업로드한다.
- GPU buffer는 월드 사분면 기준으로 나누지 않고, physical slot을 4개의 page buffer에 나눠 담는 방식으로 관리한다.
- physical slot 번호는 하나의 전역 번호로 관리하고, 실제 SSBO 접근 시에만 `pageIndex`와 `slotInPage`로 나눈다.
- 필요 없어진 청크는 GPU buffer 내용을 지우지 않고 CPU의 slot 소유권만 해제한 뒤, 이후 새 청크가 같은 physical slot을 덮어쓴다.
- raymarch shader는 월드 청크 좌표를 직접 GPU map에서 찾지 않고, 현재 로딩 그리드 기준 logical slot과 slotMap을 통해 physical slot을 찾는다.
- 현재 단계에서는 full dense capacity를 가진 fixed page buffer를 사용하되, 실제 업로드는 dense 서브청크 단위로 제한한다.
- 지형 렌더링은 현재 raymarch 방식을 기본 경로로 사용하고, F6 렌더링 전환 기능은 제거한다.
- 청크 로딩 범위는 `config/world.json`의 `loadGridScale`로 조절하고, 빌드 없이 값을 바꿀 수 있게 한다.
- `loadGridScale = n`이면 한 변이 `16n`개 청크인 정사각형을 런타임 로딩 범위로 사용한다.
- 로딩 범위 중심은 플레이어가 속한 `2 x 2` 청크 그룹으로 잡는다.
- 플레이어가 현재 `2 x 2` 청크 그룹 밖으로 나갈 때 로딩 범위를 갱신한다.
- 청크는 로딩 단위로 유지하고, 런타임 렌더링 최적화와 편집 dirty 처리는 서브청크 단위를 기준으로 한다.
- 저장 파일 양식은 아직 정하지 않고, 현재 작업에서는 런타임/GPU 데이터 구조만 다룬다.
- raymarch cell은 `uint32_t` packed cell로 표현한다.
- packed cell의 하위 16비트는 block ID로 사용하고, 상위 16비트는 추후 fluid ID용으로 예약한다.
- 유체 실제 구현은 아직 진행하지 않고, 현재 packed cell의 fluid 값은 `0`으로 둔다.
- raymarch storage buffer는 GPU shader read 성능을 위해 `DEVICE_LOCAL` 메모리에 배치하고, CPU 데이터 업로드는 staging buffer를 사용한다.
- raymarch 지형 데이터는 서브청크 metadata와 dense cell buffer로 나누어 관리한다.
- 서브청크는 `EMPTY`, `UNIFORM`, `DENSE` 상태로 구분한다.
- `EMPTY` 서브청크는 cell 데이터를 저장하지 않고 raymarch 중 다음 서브청크 경계까지 건너뛴다.
- `UNIFORM` 서브청크는 cell 하나만 저장하고, `DENSE` 서브청크만 `16 x 16 x 16` cell 데이터를 가진다.
- raymarch에서 청크/서브청크 경계에 정확히 닿은 레이는 진행 방향 기준으로 경계 안쪽으로 약간 밀어 넣어 셀을 판정한다.
- 블록 texture array에만 mipmap을 적용하고, UI, 폰트, 하늘, 플레이어 텍스처에는 적용하지 않는다.
- raymarch 블록 텍스처 mip level은 자동 LOD 대신 ray hit 거리 기준 수동 `textureLod`로 선택한다.
- mip level 전환 기준은 64블록 단위로 한다.
- 현재 블록 렌더링에는 임시 방향성 밝기 계산을 적용하지 않는다.
- 그림자와 시간대 조명은 추후 별도 조명 모델로 구현한다.
- ambient occlusion은 추가 레이를 쏘지 않고 주변 복셀 점유 여부만 조회하는 방식으로 처리한다.
- ambient occlusion은 조명 방향 계산이 아니라 블록 근접 가림 표현으로 취급한다.
- terrain raymarch 내부 렌더 해상도는 테스트 기준으로 swapchain의 0.75배를 사용한다.
- terrain 업스케일은 별도 linear sampler로 처리하고, 화면 UI 계층은 원해상도를 유지한다.
- 기본 월드 지형은 월드 `X/Z` 좌표를 입력으로 하는 사인/코사인 높이맵을 사용한다.
- 현재 높이맵 지형의 높이 범위는 `Y 120~140`이고, 생성되는 고체 블록은 모두 `rock`으로 둔다.
- 높이맵은 월드 좌표 기준으로 계산해 청크 로딩 범위가 이동해도 같은 위치의 지형 높이가 유지되도록 한다.
- CPU raymarch 청크 데이터는 청크 좌표 기준으로 캐시한다.
- 현재 GPU buffer 업로드는 전체 로딩 범위를 조립해 다시 올리는 방식을 유지한다.
- 청크 캐시는 현재 `loadGridScale` 로딩 범위 밖으로 나간 청크 데이터를 제거한다.
- 게임은 Vulkan과 C++로 개발한다.
- C++ 표준은 C++20을 기준으로 한다.
- 빌드는 CMake와 Ninja를 사용한다.
- 창 생성과 입력 처리는 GLFW를 사용한다.
- Vulkan SDK는 로컬 설치본을 사용한다.
- GLFW는 Windows 바이너리 배포본을 받아서 사용하는 방식을 기준으로 한다.
- 최종 목표는 마인크래프트류 샌드박스 복셀 게임으로 잡는다.
- 첫 목표는 CMake/Ninja 프로젝트 골격, GLFW 창 생성, Vulkan instance 생성, validation layer 연결, 빈 화면 출력 준비까지로 잡는다.
- 기능을 한 파일에 몰아넣지 않고 역할에 맞게 분리해서 개발한다.
- 구현은 현재 요구사항을 만족하는 가장 단순한 형태로 진행한다.
- 파일, 클래스, 함수는 역할과 책임 기준으로 분리하되, 분리를 위한 분리는 하지 않는다.
- 새 추상화, 공용 시스템, 외부 라이브러리는 필요성이 명확할 때만 추가한다.
- 미래에 필요할 것 같은 기능은 지금 구현하지 않는다.
- 동쪽은 월드 `X+` 방향으로 정의한다.
- 위쪽은 월드 `Y+` 방향으로 정의한다.
- 북쪽은 월드 `Z+` 방향으로 정의한다.
- 블록 좌표 `(x, y, z)`는 해당 블록의 아래면 중앙 위치를 기준으로 한다.
- 초기 해 위치는 정동쪽 수평선 방향으로 둔다.
- 초기 달 위치는 해의 정반대편 방향으로 둔다.
- 해와 달 이미지는 `assets/textures/sky/Sun.png`, `assets/textures/sky/Moon.png`를 사용한다.
- 화면 중앙 크로스헤어는 `assets/textures/ui/Crosshair.png`를 사용한다.
- 하늘은 월드 위치 이동값을 반영하지 않고 카메라 회전만 반영한다.
- 디버그 텍스트는 `assets/fonts/VCR_OSD_MONO.ttf`를 사용한다.
- 디버그 텍스트는 실행 시 기본 표시 상태로 시작한다.
- FPS, 위치, 시야 방향 표시는 초당 20회 갱신한다.
- 디버그 텍스트는 흰 글씨와 검정 테두리로 표시한다.
- 우상단 디버그 정보는 버전, CPU, GPU, Vulkan API 버전, 드라이버 버전, 해상도 순서로 표시한다.
- 해상도 표시는 `RESOLUTION: 가로 x 세로` 형식으로 표시한다.
- 좌하단 디버그 영역은 병목 확인용 임시 지표를 표시하는 공간으로 사용한다.
- 좌하단 프로파일링 지표에는 제목을 붙이지 않는다.
- 현재 좌하단 청크 갱신 프로파일링 지표는 `CHUNK UPDATE`, `WORLD BUILD`, `GPU UPLOAD`, `STALL` 네 줄만 표시한다.
- 2GB 고정 dense GPU buffer 실험 전에는 `MAX SSBO`로 Vulkan `maxStorageBufferRange`를 확인한다.
- 청크 크기는 `X 16 * Y 512 * Z 16`으로 정의한다.
- 청크는 로딩과 언로딩 단위로 사용한다.
- 청크 하나는 Y축 방향으로 `16 * 16 * 16` 크기의 서브청크 32개로 구성한다.
- 서브청크는 메싱 단위로 사용한다.
- 각 블록 셀은 블록 ID와 유체 ID를 함께 가진다.
- 블록 ID와 유체 ID는 각각 65536가지 값을 표현할 수 있도록 `uint16` 범위를 사용한다.
- 블록 ID `0`은 `air`로 예약하며 JSON 블록 정의에 명시한다.
- 초기 블록 ID는 `1 rock`, `2 grass`, `3 dirt`, `4 sand`, `5 sandstone`, `6 mud`, `7 clay`, `8 trunk`, `9 leaves`, `10000 plant`, `65535 bedrock`으로 정의한다.
- 블록 정의 JSON은 `id`, `name`, `renderType`, `textures`를 명시한다.
- 블록 텍스처 값은 확장자 없는 이름으로 작성하고, 실제 파일은 `assets/textures/block/{name}.png`로 해석한다.
- 유체 데이터는 청크 데이터 구조에 포함하지만, 초기 구현에서는 렌더링, 메싱, 시뮬레이션에 사용하지 않는다.
- 유체 데이터의 상위 9비트는 유체 종류, 하위 7비트는 유체량으로 사용한다.
- 유체 없음은 `type = 0`, `amount = 0`으로 표현한다.
- 유체 있음은 `type = 1~511`, `amount = 1~100`으로 표현한다.
- `amount = 100`은 가득 찬 상태를 의미한다.
- `amount = 101~127`은 예약값으로 사용하지 않는다.
- 블록 편집 시 변경 블록이 속한 서브청크를 dirty 처리하고, 경계 블록이면 인접 서브청크도 dirty 처리한다.

## Issues
- 청크 생성 속도 개선을 위해 `buildRaymarchChunk()` 내부 SIMD 최적화를 추후 검토한다.
- 현재 청크 생성 전용 worker queue는 유지하되, 추후 범용 job system으로 확장할지 검토한다.

## Summary
- 문서 기록 방식과 프로젝트의 기본 개발 방향을 정리했다.
