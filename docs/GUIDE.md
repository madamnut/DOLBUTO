# Development Log Guide

이 문서는 개발 기록을 작성하는 기준을 정리한다.

## Project

- 개발 언어는 C++이며, C++20을 기준으로 한다.
- 그래픽 API는 Vulkan을 사용한다.
- 빌드 시스템은 CMake와 Ninja를 사용한다.
- 창 생성과 입력 처리는 GLFW를 사용한다.
- 최종 목표는 마인크래프트류 샌드박스 복셀 게임이다.
- Vulkan SDK는 로컬 설치본을 사용한다.
- GLFW는 Windows 바이너리 배포본을 받아서 사용하는 방식을 기준으로 한다.

## Initial Goal

- CMake/Ninja 기반 프로젝트 골격을 만든다.
- GLFW 창을 생성한다.
- Vulkan instance를 생성한다.
- validation layer를 연결한다.
- 빈 화면 출력 준비까지를 첫 목표로 한다.

## Development Principles

- 기능을 한 스크립트나 한 파일에 몰아넣지 않는다.
- 렌더링, 엔진 코어, 월드, 복셀, 입력, 리소스, 플랫폼 처리 등은 역할에 맞게 분리한다.

## Implementation Rules

- 구현 전에 관련 파일과 기존 구조를 먼저 확인한다.
- 현재 요구사항을 만족하는 가장 단순한 형태로 구현한다.
- 파일, 클래스, 함수는 역할과 책임 기준으로 분리한다.
- 한 파일에 여러 책임을 몰아넣지 않는다.
- 단, 분리를 위한 분리는 하지 않는다.
- 새 추상화, 공용 시스템, 외부 라이브러리는 필요성이 명확할 때만 추가한다.
- 미래에 필요할 것 같은 기능은 지금 구현하지 않는다.
- 구현 후에는 현재 목표에 필요한 변경인지, 더 단순하게 만들 수 있는지 확인한다.

## Directory

버전별 개발 기록은 `docs/{version}/` 아래에 날짜별 Markdown 파일로 작성한다.

```text
docs/
  GUIDE.md
  0.0.0.0/
    YYYY-MM-DD.md
```

예시:

```text
docs/0.0.0.0/2026-05-01.md
```

## File Rule

- 파일명은 `YYYY-MM-DD.md` 형식을 사용한다.
- 작업이 없는 날은 파일을 만들지 않는다.
- 같은 날짜에 여러 작업을 했더라도 하나의 파일에 누적한다.
- 버전이 변경되면 새 버전 폴더를 만들고 같은 규칙을 따른다.

## Daily Format

날짜별 기록은 아래 형식을 기본으로 사용한다.

```md
# YYYY-MM-DD

## Work Log
- 작업하면서 진행한 내용을 시간순으로 적는다.

## Decisions
- 구조, 기술, 방향성에 대한 결정을 적는다.
- 결정 이유가 있으면 함께 적는다.

## Issues
- 막힌 점, 버그, 나중에 확인할 내용을 적는다.

## Summary
- 작업 마무리 시점에 하루 내용을 짧게 요약한다.
```

## Section Rules

### Work Log

작업 중 계속 갱신하는 영역이다.

- 구현한 기능
- 수정한 코드
- 조사한 내용
- 테스트하거나 확인한 내용

### Decisions

나중에 다시 봤을 때 이유를 알아야 하는 판단을 기록한다.

- 엔진 구조
- 렌더링 방식
- 외부 라이브러리 선택
- 빌드 시스템 방향
- 코드 스타일이나 프로젝트 규칙

### Issues

당장 해결하지 못했거나 추적이 필요한 내용을 기록한다.

- 버그
- 빌드 문제
- 성능 문제
- 조사 필요 항목
- 보류한 작업

### Summary

하루 작업을 마무리할 때 작성한다.

- 1~3줄 정도로 짧게 작성한다.
- 세부 내용은 `Work Log`, `Decisions`, `Issues`에 남긴다.
- 작업 중에는 비워둬도 된다.
